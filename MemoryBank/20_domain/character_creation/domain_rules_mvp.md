# Domain Rules MVP

## 1. Цель MVP

MVP покрывает только минимальный сценарий создания персонажа 1 уровня, достаточный для:
- выбора имени
- выбора `Ancestry`
- применения ancestry-based boosts и flaws
- выбора обязательных свободных boosts, если они предусмотрены ancestry
- сохранения персонажа и отображения его итоговых характеристик

В MVP не входят:
- background
- class
- class features
- equipment
- навыки
- spells
- deity
- alignment
- age
- gender
- ancestry feats
- расчёт боевых параметров кроме тех, которые уже прямо следуют из `AbilityScores` и `Ancestry`

## 2. Канонический поток MVP

Текущий процесс создания персонажа состоит из следующих обязательных шагов:

1. Пользователь вводит имя персонажа.
2. Пользователь выбирает `Ancestry`.
3. Система применяет фиксированные ancestry boosts.
4. Система применяет ancestry flaws.
5. Пользователь выбирает все обязательные ancestry free boosts.
6. Система сохраняет персонажа с итоговыми `AbilityScores`.

Порядок шагов внутри UI может быть адаптирован, но доменный результат обязан быть тем же.

## 3. Стартовое состояние персонажа

При создании персонажа:
- каждая из шести характеристик начинается со значения `10`
- `Modifier` каждой характеристики вычисляется как `(Value - 10) / 2`
- персонаж не может быть создан без привязки к аккаунту
- персонаж не может быть создан без имени
- персонаж не может быть создан без выбранного `Ancestry`

## 4. Правила имени

Имя персонажа:
- обязательно
- не может состоять только из пробелов
- должно сохраняться в обрезанном виде без ведущих и замыкающих пробелов

Если имя невалидно, команда создания должна завершаться доменной ошибкой валидации.

## 5. Правила характеристик

В MVP действуют следующие правила:

- повышение характеристики увеличивает значение на `2`
- понижение характеристики уменьшает значение на `2`
- характеристика не может повышаться одной и той же пачкой boosts дважды, если эти boosts выдаются одновременно и должны быть распределены по разным характеристикам
- итоговый `Modifier` не хранится отдельно, а вычисляется из значения

Для текущего MVP верхний предел `+4 modifier` и механика partial boosts не реализуются, потому что процесс ограничен стартовыми ancestry boosts и не включает background, class и четыре финальных boosts.

## 6. Правила выбора Ancestry

В рамках текущего проекта `Ancestry` определяет:
- набор стартовых boosts
- набор стартовых flaws
- `BaseHitPoints`
- `Size`
- `BaseSpeed`
- `Darkvision`
- `LowLightVision`

Поддерживаемые ancestry в MVP:
- `Dwarf`
- `Elf`
- `Gnome`
- `Goblin`
- `Halfling`
- `Human`

## 8. Правила free boosts

Free boosts являются обязательным шагом, если ancestry содержит хотя бы один free boost slot.

Действуют следующие инварианты:
- количество выбранных free boosts должно точно совпадать с количеством free boost slots у ancestry
- один и тот же `AbilityType` нельзя указать дважды в одном выборе free boosts
- free boost нельзя назначить в характеристику, уже повышенную fixed boost той же ancestry

Следствие:
- для `Human` пользователь обязан выбрать ровно `2` разные характеристики
- для `Dwarf`, `Elf`, `Gnome`, `Goblin`, `Halfling` пользователь обязан выбрать ровно `1` характеристику

Рекомендация для API:
- `freeBoosts` всегда передавать явно
- если у ancestry нет free boosts, передавать пустой список

## 9. Правила повторного выбора ancestry

Если в процессе чернового редактирования пользователь меняет ancestry до финального сохранения:
- все ранее применённые fixed boosts старой ancestry должны быть откатаны
- все ранее применённые flaws старой ancestry должны быть откатаны
- все ранее выбранные free boosts старой ancestry должны быть откатаны
- после этого применяется новая ancestry и её правила

Это обязательное инвариантное поведение доменной модели.

## 10. Что должно храниться в агрегате

Для MVP в агрегате персонажа должны храниться как минимум:
- владелец персонажа
- имя
- выбранный `Ancestry`
- итоговые `AbilityScores`
- список применённых free boosts

Дополнительно могут храниться денормализованные ancestry-derived поля, если это нужно для чтения, но источником истины для правил выбора остаётся `Ancestry`.

## 11. Что должно вычисляться

На основе итогового состояния персонажа должны вычисляться:
- `Characteristic.Modifier`
- общая Масса переносимых предметов, нагрузка и пороги перегрузки из значения `Strength`
- derived-поля ancestry, если они нужны для DTO

Правило: вычисляемые параметры не должны вводиться пользователем вручную, если их можно однозначно получить из доменных данных.

## 12. API-инварианты

Контракт создания персонажа должен проверять:
- пользователь аутентифицирован
- `name` заполнен
- `ancestryType` задан и входит в поддерживаемый список
- `freeBoosts` не нарушают инварианты ancestry

Сервер обязан сам вычислять итоговые `AbilityScores`.
Клиент не должен отправлять готовые значения характеристик как источник истины.

## 13. Инварианты для тестов

Обязательные сценарии тестирования:
- персонаж создаётся со всеми характеристиками `10`, если ещё не применены ancestry rules
- `Dwarf` даёт `Constitution +2`, `Wisdom +2`, `Charisma -2` и один free boost
- `Human` требует ровно два разных free boosts
- free boost нельзя направить в характеристику из fixed boosts той же ancestry
- смена ancestry откатывает предыдущие boosts и flaws
- итоговый modifier корректно вычисляется из итогового score

## 14. Решения, зафиксированные для проекта

Для текущего этапа фиксируются следующие проектные решения:
- `Ancestry` является основной доменной сущностью выбора народа
- источник истины по стартовым boosts и flaws находится в доменной модели, а не в UI
- API принимает решения пользователя, но не доверяет клиенту итоговые характеристики
- документация и будущая реализация должны постепенно вытеснить доменную двусмысленность между `Race` и `Ancestry`
