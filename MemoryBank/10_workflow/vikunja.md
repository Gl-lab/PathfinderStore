# Vikunja Workflow

## Когда читать

Читать перед любым созданием, обновлением, закрытием или анализом задач Vikunja.

## Контекст

Для ведения задач используется MCP Vikunja (`@0xk3vin/vikunja-mcp`).

Конфигурация:

- конфиг: `.mcp.json` в корне проекта;
- Vikunja URL: `http://192.168.1.85:3456`;
- проект: `Pathfinder`;
- project ID: `2`.

## Главное правило

Перед созданием новой задачи всегда сначала проверять существующие задачи проекта:

```text
mcp__vikunja__tasks_list { projectId: 2 }
```

Это нужно, чтобы не создавать дубли.

## Проблема как обязательная часть задачи

Каждая новая или существенно переформулированная задача должна явно описывать проблему, которую она решает. Список действий без проблемного контекста недостаточен: по нему нельзя проверить, устраняет ли реализация исходную потребность.

В описании задачи обязательно фиксировать:

- **Проблема** — наблюдаемое нежелательное состояние или недостающая возможность;
- **Последствия** — кому и чему мешает проблема;
- **Ожидаемый результат** — какое состояние должно быть после решения;
- **Критерии готовности** — проверяемые признаки того, что проблема действительно устранена.

Критерии готовности должны проверять ожидаемый результат, а не только факт выполнения технических действий. Например, наличие миграции само по себе не доказывает устранение рассогласования данных; нужен тест или сценарий, подтверждающий согласованное состояние системы.

Если задача содержит только формулировку «добавить», «реализовать» или перечень файлов и компонентов, перед началом реализации нужно дополнить её проблемой и ожидаемым результатом.

## Безопасное обновление задачи

Важно: в текущем MCP Vikunja уже дважды наблюдался опасный эффект — частичный `tasks_update` с полями вроде `done` или `percentDone` возвращал задачу с пустым `description`.

Чтобы не потерять описание задачи:

1. Перед любым `tasks_update` сначала перечитать задачу через `tasks_list` и сохранить текущие `title`, `description`, `done`, `percent_done`.
2. При обновлении статуса, процента, заголовка или других полей всегда передавать обратно текущий `description`, даже если текст описания не меняется.
3. После `tasks_update` перечитать задачу и проверить, что:
   - `description` не пустой;
   - кириллица отображается корректно;
   - изменённые поля применились.
4. Если описание всё же обнулилось, сразу повторить `tasks_update` с сохранённым `description`.

Практический минимум для status/progress update:

```json
{
  "taskId": 54,
  "done": false,
  "percentDone": 0.5,
  "description": "<current description>"
}
```

## Как читать задачу по номеру

Если пользователь ссылается на номер вида `#23` или говорит “задача 23”, сначала трактовать это как `index` / `identifier`, а не как внутренний `id`.

Порядок:

1. Получить список задач проекта через Vikunja.
2. Найти задачу по `identifier` или `index`.
3. Только после этого, если нужен контекст реализации, искать связанные файлы в репозитории.

Не ходить в репозиторий искать `task 23`, `#23`, `задача 23` и похожие строки, пока не была проверена сама карточка в Vikunja.

## Кодировка

При создании и обновлении задач с кириллицей использовать только UTF-8 end-to-end.

После `mcp__vikunja__tasks_create` и `mcp__vikunja__tasks_update` обязательно перечитать задачу или список задач и убедиться, что кириллица отображается корректно.

Если ответ Vikunja читается как `Ð...`, считать это проблемой декодирования ответа, а не содержимого задачи.

Задачи `#17`-`#21` считаются зафиксированным антипримером по повреждённой кодировке.

## Форматирование описаний и комментариев

Текущий интерфейс Vikunja отображает Markdown, переданный через MCP, как обычный текст. Поэтому для форматированных описаний задач и комментариев через MCP нужно передавать HTML.

Использовать минимальный набор тегов:

- `<p>` — абзац;
- `<h2>` и `<h3>` — заголовки разделов;
- `<ul>`, `<ol>` и `<li>` — списки;
- `<code>` — пути, имена типов, команды и короткие фрагменты кода;
- `<strong>` — короткий смысловой акцент;
- `<pre><code>...</code></pre>` — многострочный код или команда.

Не передавать через MCP Markdown-конструкции `##`, `-`, обратные кавычки и fenced code blocks в расчёте на их автоматический рендеринг.

Рекомендуемая структура описания задачи:

```html
<h2>Проблема</h2>
<p>Наблюдаемое нежелательное состояние и его последствия.</p>
<h2>Ожидаемый результат</h2>
<p>Состояние системы после устранения проблемы.</p>
<h2>Что нужно сделать</h2>
<ul>
  <li>Первый пункт.</li>
  <li>Второй пункт с <code>именем сущности</code>.</li>
</ul>
<h2>Критерии готовности</h2>
<ul>
  <li>Проверяемый результат.</li>
</ul>
```

Рекомендуемая структура статусного комментария:

```html
<h2>Статус</h2>
<p><strong>Состояние:</strong> в работе.</p>
<h3>Что сделано</h3>
<ul>
  <li>Завершённый результат.</li>
</ul>
<h3>Что осталось</h3>
<ul>
  <li>Следующий шаг.</li>
</ul>
<h3>Проверки</h3>
<ul>
  <li>Выполненная проверка и её результат.</li>
</ul>
```

Для итогового комментария заменить блок `Что осталось` на явный результат завершения. После записи обязательно перечитать задачу или комментарии и проверить, что HTML сохранён, кириллица не повреждена, а интерфейс отрисовывает разметку.

## Известная проблема локального MCP

Claude Code может передавать параметры MCP-инструментов как строки.

Локальный `@0xk3vin/vikunja-mcp` использует строгие Zod-типы, поэтому числа и булевы значения могут отвергаться с ошибкой `-32602`.

Патч применяется к глобально установленному пакету по пути:

```text
%APPDATA%\npm\node_modules\@0xk3vin\vikunja-mcp\dist\index.js
```

Проверенное изменение:

- `z.number()` заменить на `z.coerce.number()`;
- `z.boolean()` заменить на `z.coerce.boolean()`.

Если после обновления пакета инструменты снова перестали принимать числа и булевы значения, патч нужно применить повторно.

## Проверенный контракт инструментов

- Создание задачи: `PUT /projects/{id}/tasks` через `mcp__vikunja__tasks_create`.
- Обновление задачи: `POST /tasks/{id}` через `mcp__vikunja__tasks_update`.
- Связь подзадачи: `PUT /tasks/{id}/relations` через `mcp__vikunja__task_relations_add`.

Если после обновления пакета поведение снова расходится с этим контрактом, сначала перепроверить локальный `vikunja-mcp`, и только потом создавать или обновлять задачи.

## Основные инструменты

- `mcp__vikunja__ping` — проверить подключение.
- `mcp__vikunja__tasks_list` — список задач.
- `mcp__vikunja__tasks_create` — создать задачу.
- `mcp__vikunja__tasks_update` — обновить задачу.
- `mcp__vikunja__tasks_delete` — удалить задачу.
- `mcp__vikunja__task_relations_add` — добавить связь между задачами.

## Правило типов

Параметры передавать как нативные типы, а не строки:

```json
{
  "projectId": 2,
  "done": false
}
```

## Анти-паттерны

- Не делать предположение, что URL `/tasks/23` вернёт задачу `#23`: это внутренний `id`, а не пользовательский номер задачи.
- Не использовать прямой HTTP API как первый шаг, если задача решается через штатный Vikunja workflow.
- Не комбинировать чтение задачи из Vikunja с параллельным широким поиском по репозиторию “на всякий случай”.
- Не запускать поиск по всему дереву без исключения служебных каталогов.
- Не принимать повреждённую кодировку в ответах Vikunja за содержимое задачи.

## Приоритет инструментов в этой среде

- Для поиска и просмотра файлов сначала использовать PowerShell cmdlets: `Get-ChildItem`, `Select-String`, `Get-Content`.
- `rg` использовать только если он доступен и запускается без ошибок.
