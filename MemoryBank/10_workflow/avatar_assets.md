# Avatar Assets Workflow

## Когда читать

Читать перед генерацией, конвертацией или подключением тематических аватаров персонажей.

## Главное правило

Полный набор из 192 изображений не является обязательным условием работы системы. Аватары добавляются независимыми партиями по доступному бюджету и могут быть сгенерированы позднее.

Для сочетаний, которых ещё нет в runtime-каталоге, backend назначает `avatar.system.unknown`. Добавление новых файлов не переназначает аватары ранее созданных персонажей.

Успешно сгенерированный исходник используется без визуального QA: не открывать изображение для оценки, не создавать preview круглого crop или `48×48` и не запускать повторную генерацию из-за внешнего вида. Сохраняются только автоматические технические проверки файла, metadata, каталога и тестов.

Запись в `AvatarCatalog` добавляется только одновременно с технически корректным WebP-файлом. Нельзя регистрировать путь к отсутствующему или повреждённому изображению.

## Источники истины

- [матрица ID и metadata](../90_research/avatar_asset_matrix.md) определяет `AvatarId`, путь, ancestry, class, gender и variant;
- [art direction и production template](../90_research/avatar_generation_prompts.md) определяют признаки ancestry и class, композицию, варианты и negative prompt;
- [runtime-каталог](../../CharacterManagement.Application/Avatars/AvatarCatalog.cs) содержит только подключённые ассеты;
- [тесты каталога](../../CharacterManagement.Infrastructure.Tests/AvatarSelectorTests.cs) фиксируют покрытые и непокрытые сочетания;
- [заметки партий](../90_research/avatar_batches/) фиксируют происхождение и техническую проверку файлов.

## 1. Выбрать строки матрицы

1. Открыть [матрицу ассетов](../90_research/avatar_asset_matrix.md).
2. Найти строки, для которых ещё нет WebP в `pathfinder.frontend/public/avatars/pc/`.
3. Сохранить из каждой строки без изменений:
   - `AvatarId`;
   - относительный URL;
   - ancestry;
   - `ClassId`;
   - gender;
   - variant.
4. Не перенумеровывать строки и не переиспользовать ID изображения для другого сочетания.

Партия может состоять даже из одного ассета. Незавершённая комбинация ancestry + class безопасна: отсутствующий gender или variant продолжает использовать заглушку.

## 2. Сформировать production prompt

Промт формируется на английском по шаблону из [пакета промтов](../90_research/avatar_generation_prompts.md).

Обязательные подстановки:

1. `Primary request` — точные ancestry, class, gender и variant из матрицы.
2. `Subject ancestry` — визуальные признаки ancestry из проверенной таблицы, без признаков других ancestry.
3. `Character design` — самостоятельный фенотип, палитра, волосы и экипировка, отличающие результат от соседних вариантов.
4. `Class expression` — ровно один хорошо читаемый классовый маркер из профиля class.
5. `Composition/framing` и `Lighting/mood` — правила выбранного variant.
6. Общие constraints и полный negative prompt.

Постоянные требования:

- `Use case: stylized-concept`;
- один взрослый персонаж;
- квадратный портрет `1024×1024`, по грудь или плечи;
- лицо, макушка, подбородок, уши, ancestry-признаки и классовый маркер внутри центральных 70%;
- безопасный круглый crop и читаемость при `48×48`;
- ancestry является главным визуальным акцентом, один классовый маркер — вторичным;
- простой размытый fantasy-neutral фон;
- оригинальная стилизованно-реалистичная fantasy-иллюстрация;
- без текста, логотипов, рамок, UI, официальных персонажей, копирования иллюстраций и имитации живущего художника;
- без ребёнка или childlike adult, современных предметов, лишних персонажей и смешения ancestry-признаков.

### Различия вариантов

Variant 1:

- лёгкий разворот влево;
- спокойное уверенное выражение;
- мягкий тёплый ключевой свет;
- аккуратная экипировка;
- наиболее узнаваемый классовый предмет у плеча.

Variant 2:

- лёгкий разворот вправо;
- настороженное или сосредоточенное выражение;
- нейтрально-холодный ключевой свет;
- более походная экипировка;
- другой допустимый фенотип и альтернативный классовый маркер.

## 3. Сгенерировать исходник

Допустим встроенный генератор Codex или внешний генератор изображений. Инструмент генерации не меняет требования к промту.

Целевой исходник:

- квадратный PNG `1024×1024` или больше;
- непрозрачное RGB/RGBA-изображение;
- без встроенного текста, рамки и watermark.

Исходные PNG являются промежуточными мастер-файлами и по умолчанию не добавляются в Git. Их можно временно передать через `tempinfo/` или внешний каталог. Не удалять предоставленный пользователем исходник без явного запроса.

Сбой генератора не блокирует функцию аватаров и не требует добавлять placeholder под тематическим ID. Нужно оставить строку матрицы непокрытой и повторить генерацию позднее.

## 4. Выполнить техническую проверку исходника

После единственного generation call не выполнять визуальный QA. Не использовать `view_image`, не создавать круглый crop или preview `48×48`, не оценивать соответствие внешнего вида промту и не перегенерировать результат по визуальным причинам.

Автоматически проверить только:

1. Файл существует и открывается библиотекой изображений.
2. Изображение квадратное и имеет размер не меньше `1024×1024`.
3. Режим изображения — непрозрачный RGB или RGBA без фактической прозрачности.

При технической ошибке не подключать файл. Внешний вид не является причиной отклонения или повторной генерации.

## 5. Конвертировать сгенерированный PNG в WebP

Для конвертации используется Python с Pillow. Команда запускается из корня репозитория. Заменить пути и номер файла на выбранную строку матрицы.

```powershell
@'
from PIL import Image

source = r"tempinfo/21.png"
target = r"pathfinder.frontend/public/avatars/pc/000021.webp"

with Image.open(source) as image:
    image = image.convert("RGB")
    image = image.resize((512, 512), Image.Resampling.LANCZOS)
    image.save(target, "WEBP", quality=88, method=6)
'@ | python -
```

Если исходник не квадратный, нельзя растягивать его до квадрата. Сначала согласовать и выполнить crop с сохранением центральной безопасной области.

Проверка результата:

```powershell
@'
from PIL import Image

path = r"pathfinder.frontend/public/avatars/pc/000021.webp"

with Image.open(path) as image:
    print(image.format, image.size, image.mode)
'@ | python -
```

Ожидаемый результат:

```text
WEBP (512, 512) RGB
```

## 6. Подключить сгенерированный ассет

Для каждой сгенерированной строки выполнить все действия в одном изменении:

1. Добавить `pathfinder.frontend/public/avatars/pc/NNNNNN.webp`.
2. Добавить `AvatarDefinition` в `AvatarCatalog` с точными ID, URL и metadata из матрицы.
3. Обновить `AvatarSelectorTests`:
   - добавить или изменить ожидаемое количество для покрытого сочетания;
   - удалить либо перенести assertion, который объявлял сочетание непокрытым;
   - сохранить проверку реально непокрытого сочетания.
4. Создать следующую заметку в `MemoryBank/90_research/avatar_batches/` с:
   - диапазоном ID;
   - metadata;
   - происхождением исходника;
   - итоговыми путями;
   - результатами технической проверки формата, размера и режима.
5. Обновить текущий объём каталога в [задачной заметке](../30_task_notes/character_gender_and_avatars.md), если граница подключённого диапазона изменилась.

Не изменять `AvatarId` уже созданных персонажей и не запускать массовое переназначение заглушек.

## 7. Проверить изменение

Минимальные проверки:

```powershell
dotnet test CharacterManagement.Infrastructure.Tests/CharacterManagement.Infrastructure.Tests.csproj --filter AvatarSelectorTests
git diff --check
git status --short
```

Также проверить:

- каждый зарегистрированный URL соответствует существующему WebP;
- каждый новый файл имеет `512×512`, формат WebP и режим RGB;
- metadata каталога дословно совпадает с матрицей;
- заметка партии использует только относительные пути;
- в коммит не попали временные PNG и посторонние файлы.

## Возобновление работы позднее

Чтобы продолжить после паузы:

1. Найти первую строку матрицы без WebP и записи в runtime-каталоге.
2. Проверить последние заметки партий, чтобы не повторить уже подключённый ассет.
3. Сформировать промт по правилам раздела 2.
4. Пройти полный цикл generation → technical validation → WebP → catalog → tests → batch note.

Наличие пропусков допустимо, если каждый зарегистрированный ассет согласован с матрицей и физически существует.
