# Пол персонажа и расширяемая система аватаров

## Статус документа

Согласованное ТЗ и план реализации новой функции. Документ фиксирует продуктовые и технические решения до начала разработки.

Vikunja: parent-задача `#47`, этапы `#48`–`#52`.

Текущий статус: этапы `#48`–`#51` реализованы и прошли самостоятельное ревью; этап `#52` начат — первые пять ассетов сгенерированы, приняты и подключены к runtime-каталогу.

## Проблема

В текущем character creation flow отсутствует пол персонажа. В списке персонажей вместо портрета показывается первая буква имени, а постоянного аватара у персонажа нет.

Полный набор тематических изображений нельзя считать обязательным условием запуска: для текущего каталога требуется 192 изображения, и их одновременная генерация может превысить доступный бюджет.

## Ожидаемый результат

- Новый персонаж обязательно получает мужской или женский пол на первом экране мастера создания.
- Старые записи сохраняются с внутренним значением `NotSpecified` и требуют выбора пола при первом открытии.
- Каждый персонаж имеет сохранённый стабильный `AvatarId`.
- Backend случайно выбирает доступный аватар по параметрам нового персонажа или назначает системную заглушку.
- Назначенный аватар, включая заглушку, не меняется автоматически.
- Тематические изображения можно добавлять партиями без изменения доменной модели и существующих персонажей.

## Согласованный функциональный scope

### Пол персонажа

Используется enum `CharacterGender`:

| Значение | Назначение |
|---|---|
| `NotSpecified = 0` | Только внутреннее состояние старых записей |
| `Male = 1` | Мужской пол |
| `Female = 2` | Женский пол |

Правила:

- UI создания предлагает только `Male` и `Female`.
- Пол обязателен для всех новых персонажей.
- Пол располагается на первом экране мастера рядом с именем, концепцией и возрастом.
- Клиентская и серверная валидация отклоняют `NotSpecified` при создании.
- Существующие записи после миграции получают `NotSpecified`.
- При первом открытии старого персонажа UI блокирует обычную работу с карточкой и требует выбрать `Male` или `Female`.
- Отдельная серверная операция разрешает переход только из `NotSpecified` в `Male` или `Female`.
- Повторное изменение пола в текущем scope запрещено.

### Аватар персонажа

У персонажа хранится стабильный непрозрачный идентификатор `AvatarId`. Идентификатор не кодирует сочетание параметров и не используется для восстановления метаданных.

Примеры:

- `avatar.system.unknown` — общая системная заглушка;
- `avatar.pc.000001` — тематический аватар Player Core.

Путь к файлу и условия выбора принадлежат каталогу аватаров, а не aggregate персонажа.

### Системная заглушка

`avatar.system.unknown` — полноценное постоянное назначение, а не временное вычисляемое состояние.

Заглушка назначается:

- всем существующим персонажам во время миграции независимо от их дальнейшего выбора пола;
- новому персонажу с `NotSpecified`, если такой запрос всё же поступил из legacy/internal flow;
- новому персонажу, для сочетания параметров которого нет доступных тематических аватаров;
- при защитном fallback, если каталог не смог вернуть валидный тематический вариант.

Выбор пола старым персонажем не меняет `avatar.system.unknown`. Появление новых изображений в каталоге также не изменяет ранее назначенные аватары.

Автоматическая замена заглушки запрещена. Будущая смена возможна только через отдельно спроектированную пользовательскую функцию или админку.

## Текущий объём каталога

Текущие ancestry:

- Dwarf;
- Elf;
- Gnome;
- Goblin;
- Halfling;
- Human.

Текущие классы:

- Bard;
- Cleric;
- Druid;
- Fighter;
- Ranger;
- Rogue;
- Witch;
- Wizard.

Целевой полный набор:

```text
6 ancestry × 8 classes × 2 genders × 2 variants = 192 avatars
```

Все 192 изображения не входят в минимальный критерий готовности системы. Генерация и подключение выполняются партиями в пределах доступного бюджета.

## Расширяемый каталог аватаров

Запись каталога должна отделять стабильный ID от критериев выбора:

```text
Id: avatar.pc.000001
AssetPath: /avatars/pc/000001.webp
Ancestry: Dwarf
ClassId: class.fighter
Gender: Male
Variant: 1
HeritageId: null
ClassSpecializationId: null
BackgroundId: null
```

Обязательные критерии версии 1:

- `Ancestry`;
- `ClassId`;
- `Gender`.

`HeritageId`, `ClassSpecializationId`, `BackgroundId` и другие будущие признаки являются метаданными расширения. Правила приоритета специализированных аватаров не входят в текущий scope и должны быть отдельно согласованы до их использования.

Добавление новых записей не должно требовать изменения формата `AvatarId` или данных уже созданных персонажей.

## Правила назначения

Назначение выполняется backend один раз в транзакции создания персонажа:

1. Если пол равен `NotSpecified`, выбрать `avatar.system.unknown`.
2. Найти активные записи каталога версии 1 с точным совпадением ancestry, class и gender.
3. Если найден хотя бы один вариант, случайно выбрать одну запись.
4. Если вариантов нет или каталог недоступен, выбрать `avatar.system.unknown`.
5. Сохранить выбранный `AvatarId` вместе с персонажем.

Случайный выбор должен быть вынесен за интерфейс, чтобы тесты могли использовать детерминированную реализацию.

После сохранения запрещены:

- повторный случайный выбор при чтении персонажа;
- вычисление аватара из текущих свойств вместо чтения `AvatarId`;
- автоматическая замена заглушки;
- изменение настоящего аватара при пополнении каталога.

## API и legacy flow

### Создание

`POST /api/character` принимает обязательный `gender` со значением `Male` или `Female`. `AvatarId` клиент не передаёт: его назначает только сервер.

### Чтение

Список и карточка персонажа возвращают:

- `gender`;
- `avatarId`;
- публичный `avatarPath` либо эквивалентную read-модель, достаточную для отображения ассета.

### Выбор пола для старой записи

Нужна авторизованная операция уровня персонажа, например:

```text
PUT /api/character/{characterId}/gender
```

Она:

- проверяет принадлежность персонажа текущему аккаунту;
- принимает только `Male` или `Female`;
- разрешена только при текущем `NotSpecified`;
- не меняет `AvatarId`;
- возвращает обновлённую read-модель или успешный результат, после которого UI перечитывает персонажа.

## Миграции данных

Миграции EF Core создаются только через `dotnet ef`. По этапам они добавляют обязательные колонки с безопасными значениями для старых строк:

- `Gender = NotSpecified`;
- `AvatarId = avatar.system.unknown`.

После миграции ни одна существующая запись не должна иметь `NULL` в этих полях.

## Требования к изображениям

### Формат

- исходник генерации: `1024×1024`;
- ассет приложения: `512×512 WebP`, ориентир качества 85%;
- композиция безопасна для круглого crop;
- лицо и основные признаки находятся в центральных 70% кадра;
- целевые размеры отображения: `48×48` в списке и до `192×192` в карточке.

### Композиция тематического аватара

- портрет по грудь или плечи, допустим ракурс 3/4;
- лицо и признаки ancestry — главный акцент;
- одежда, экипировка и один узнаваемый признак класса — второй акцент;
- простой размытый фон;
- единый стилизованный реалистичный fantasy-стиль;
- без текста, рамок, UI-элементов и сложной сцены;
- без прямого копирования иллюстраций Archives of Nethys.

Для описания внешности ancestry и классовых признаков используется Archives of Nethys Remaster. AoN является источником правил и описаний, а не изображением для копирования.

### Композиция заглушки

- тёмный нейтральный фон;
- контрастный силуэт головы и плеч;
- отсутствуют выраженные признаки пола, ancestry, класса и экипировки;
- изображение остаётся читаемым при `48×48`.

## UI

### Мастер создания

- На первом шаге добавить обязательный выбор пола двумя локализованными вариантами.
- Ошибка обязательности должна быть видна до отправки формы.
- В запрос создания передаётся выбранный `gender`.

### Старый персонаж

- При `NotSpecified` карточка показывает модальное или отдельное обязательное состояние выбора пола.
- Закрытие выбора не должно позволять обойти требование и продолжить обычную работу с карточкой.
- После успешного выбора обновляются данные персонажа, но заглушка остаётся прежней.

### Отображение аватара

- В dashboard заменить буквенный placeholder изображением из read-модели.
- В карточке персонажа показать увеличенную версию того же изображения.
- При ошибке загрузки файла UI показывает `avatar.system.unknown`, не изменяя данные персонажа.
- Для изображения задать осмысленный `alt` на основе имени персонажа.

## Критерии готовности

- Новый персонаж нельзя создать без `Male` или `Female`.
- Существующие персонажи после миграции имеют `NotSpecified` и `avatar.system.unknown`.
- Старый персонаж может один раз выбрать `Male` или `Female`; повторная смена отклоняется.
- Выбор пола старым персонажем не меняет заглушку.
- Новый персонаж получает один из доступных точных вариантов либо `avatar.system.unknown`.
- Повторные чтения и пополнение каталога не меняют сохранённый `AvatarId`.
- Клиент не может назначить произвольный `AvatarId` через create API.
- Dashboard и карточка отображают один и тот же сохранённый аватар.
- Ошибка загрузки ассета безопасно отображает заглушку.
- Система работает при наличии только одной заглушки и не требует всех 192 изображений.
- Domain, application/infrastructure/API и frontend сценарии покрыты тестами соразмерно изменению.
