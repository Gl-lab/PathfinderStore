# Priority 9 — прогресс реализации

## Источник задач

Priority 9 определён в [ближайшем roadmap](character_creation_near_term_roadmap.md) и [целевой архитектуре кампаний](../20_domain/store/target_architecture_togaf.md).

Vikunja недоступна в текущем окружении, поэтому состояние выполнения временно фиксируется в этом документе и в отдельных Git-коммитах.

## Выполнено

### Slice 1. Campaign aggregate, жизненный цикл и роль создателя-ведущего

- добавлен отдельный bounded context `CampaignManagement` с domain/application/infrastructure слоями;
- создание кампании атомарно создаёт активное членство автора с кампанийной ролью `GameMaster`;
- роль ведущего не добавлена в глобальные роли `Secure`;
- реализован жизненный цикл `Active -> Archived` без hard delete;
- данные хранятся в отдельной схеме `campaign_management`, миграция создана через `dotnet ef`;
- API поддерживает список доступных пользователю кампаний, создание и архивирование;
- frontend поддерживает создание, просмотр статуса/роли и архивирование кампании;
- доменные, infrastructure и frontend тесты покрывают инвариант создателя-ведущего и изоляцию списка по пользователю.

### Slice 2. Членство, приглашение, выход и роли ведущего/игрока

- ведущий приглашает пользователя по имени через application-порт к пользовательскому каталогу `Secure`;
- адресат видит собственные ожидающие приглашения и может принять или отклонить каждое из них;
- принятие приглашения создаёт активную кампанийную роль `Player`, не изменяя глобальные роли пользователя;
- активный ведущий может назначить участнику дополнительную роль `GameMaster` или снять её;
- участник может выйти из кампании, но последний активный ведущий не может выйти или лишиться роли;
- изменения членства запрещены после архивирования кампании;
- API и frontend поддерживают приглашения, ответы, список участников, изменение роли ведущего и выход;
- отдельная EF migration добавляет приглашения с адресатом, автором, статусом и временными отметками.

### Slice 3. Первая партия и контролируемые персонажи

- агрегат кампании хранит коллекцию партий, но на текущем этапе допускает только одну активную партию;
- ведущий создаёт первую партию, а модель сохраняет возможность последующего расширения до нескольких партий;
- активный игрок может добавить собственного персонажа, ведущий — назначить персонажа другому активному игроку;
- application-порт к `CharacterManagement` проверяет владельца персонажа без внешнего ключа между bounded context;
- API и frontend поддерживают создание партии, выбор собственного персонажа и назначение персонажа ведущим;
- отдельная EF migration добавляет партии и назначения персонажей, включая уникальность персонажа внутри партии.

### Slice 4. Серверная политика доступа к карточке и действиям персонажа

- доступ проверяется по актуальной кампании, активной партии, членству, роли и назначению персонажа;
- назначенный игрок может просматривать карточку и выполнять обычные игровые действия своего персонажа;
- ведущий может просматривать карточки только персонажей активной партии своей кампании, но не выполняет обычные действия игрока;
- произвольный ID персонажа, роль в другой кампании и глобальная системная роль не дают доступа;
- карточка и изменение HP получили campaign-scoped API с обязательным `campaignId`, старый бесконтекстный endpoint изменения HP удалён;
- frontend открывает карточки партии в контексте кампании и показывает элементы изменения HP только назначенному игроку;
- owner-scoped создание, финализация и удаление персонажа остаются отдельным личным flow.
