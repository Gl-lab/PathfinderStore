# Исторический план переписывания frontend на Vue 3

## Статус

План выполнен и сохранён как decision record. Vue 3 + TypeScript SPA находится в `pathfinder.frontend`; legacy директории `pathfinder.vue/` в текущем рабочем дереве нет.

Реализованный frontend покрывает аутентификацию, dashboard, список и карточку персонажа, wizard creation flow, обработку ошибок и frontend tests. Дальнейшие изменения следует планировать от текущей реализации и актуальных API-контрактов, а не от фаз этого исторического плана.

## Решение

На момент решения frontend в `pathfinder.vue/` не мигрировался поэтапно внутри Vue 2. Вместо этого был создан новый SPA на Vue 3, который после функциональной приёмки стал единственным frontend приложения.

Целевой стек:

- Vue 3 + TypeScript;
- Vite;
- Vue Router 4;
- Pinia;
- Vuetify 3;
- Axios с единым API-клиентом;
- ESLint + Prettier;
- Vitest.

Причины решения: legacy frontend был небольшим, а основной пользовательский сценарий ещё развивался. Перенос Vue 2/Vuetify 2 кода с промежуточной совместимостью не снижал объём работ и сохранял бы устаревшую сборку.

## Границы первого релиза

В первый релиз нового frontend входят:

1. Вход, регистрация, хранение сессии и выход.
2. Главная страница как dashboard персонажей, а не информационный экран.
3. Список персонажей.
4. Wizard создания персонажа: имя, концепция, возраст, выбор родословной, fixed/free boosts, проверка и подтверждение.
5. Карточка персонажа и удаление персонажа, если endpoint уже доступен.
6. Единая обработка загрузки, пустых состояний и ошибок API.

Не расширять этот релиз магазином, инвентарём, каталогами способностей и другими PF2e-механиками.

## Что не переносим из Vue 2

Старые файлы оставляются в `pathfinder.vue/` только как исторический ориентир до переключения на новый frontend. Они не должны копироваться или адаптироваться в новый код.

| Старый экран или компонент | Что делал | Решение |
|---|---|---|
| `src/views/About.vue` | Информационная главная страница с декоративным изображением и сообщениями. | Удалить. Главной станет dashboard персонажей. |
| `src/views/VuetifyProduct.vue` | Список товаров с серверной пагинацией. | Не переносить: Store не в активном scope. |
| `src/views/CreateProductForm.vue` | Форма создания товара. | Не переносить: Store не в активном scope. |
| `src/components/comboBox/CategoryComboBox.vue` | Выбор категории товара из API. | Не переносить вместе со Store. |
| `src/models/Product.js` | Клиентская модель товара. | Не переносить вместе со Store. |
| `src/models/Wallet.js` | Клиентская модель кошелька. | Не переносить: в MVP нет пользовательского сценария. |
| `src/character/CharacteristicsForm.vue` | Ручная настройка шести характеристик слайдерами. | Не переносить: противоречит текущему MVP, где характеристики формируются правилами создания. |

Текущие `CharacterCreateForm.vue`, `RaceSelect.vue` и `FreeBoostsSelect.vue` описывают полезный сценарий, но также не переносятся построчно: их поведение реализуется заново в wizard-е с типами, явными состояниями и API-контрактами.

## Дизайн-концепция: «Character Ledger»

Интерфейс — спокойный персональный кабинет игрока, а не стилизация под лист персонажа или перегруженный фэнтези-сайт. Pathfinder-настроение создаётся цветом, типографикой и небольшими иллюстративными акцентами; правила и данные остаются главными.

### Принципы

- **Данные прежде декора.** Имя, родословная, характеристики и следующий шаг считываются быстрее, чем декоративные элементы.
- **Одна ключевая задача на экран.** Dashboard ведёт к персонажу или созданию; wizard ведёт к следующему решению.
- **Мягкая тематичность.** Тёмный ink/navy фон навигации, тёплый parchment фон контента, акцентный amber/copper. Не использовать текстуры, мешающие читаемости.
- **Предсказуемость.** Единые статусы loading/empty/error, одинаковые карточки и кнопки, доступный контраст и видимый focus.
- **Mobile-first.** Wizard и карточка полностью функциональны с ширины 320 px; на desktop появляется двухколоночная компоновка.

### Структура экранов

- **App shell:** слева на desktop компактная навигация; на mobile — верхняя панель и drawer. В навигации только «Персонажи», «Создать персонажа» и меню профиля.
- **Dashboard персонажей:** заголовок «Мои персонажи», краткая подсказка для пустого списка, заметная кнопка «Создать персонажа», сетка карточек. Каждая карточка показывает имя, родословную, возраст/концепцию при наличии и открывается целиком.
- **Wizard:** верхний progress stepper с 4 шагами: «Основное», «Родословная», «Бусты», «Проверка». На каждом шаге: заголовок, короткое объяснение правила, одна форма и кнопки «Назад/Далее».
- **Карточка персонажа:** крупное имя и badge родословной, затем группы данных «Концепция», «Характеристики», «Детали». Опасные действия отделены в menu/dialog.

### Базовые design tokens

Токены задаются через Vuetify theme и CSS custom properties, без произвольных hex-значений внутри компонентов:

- `surface`: тёплый светлый фон;
- `surface-variant`: фон карточек и блоков;
- `primary`: глубокий navy для навигации и основных действий;
- `secondary`: teal для статусов и второстепенных акцентов;
- `accent`: amber для прогресса и важных акцентов;
- `error`, `warning`, `success`: семантические цвета Vuetify;
- шаг сетки: 4/8 px;
- радиус карточек: 12 px;
- основной текст: системный sans-serif, display-шрифт допустим только для названий экранов после проверки кириллицы.

## Архитектура нового frontend

```text
src/
  app/             # Vuetify theme, router, providers
  api/             # axios instance, endpoints, DTO и преобразования ошибок
  features/
    auth/          # login, registration, session store
    characters/    # list, details, delete, character API
    character-creation/ # wizard, local form state, validation
  shared/          # reusable UI, constants, helpers, layout
  views/           # route-level composition only
```

Правила:

- компоненты не вызывают Axios напрямую;
- Pinia хранит только межэкранное состояние (`auth`, при необходимости cache списка); состояние wizard-а локально в feature;
- DTO API отделены от presentation-моделей;
- backend остаётся источником доменной истины, клиент проверяет только понятные пользователю предусловия;
- каждый endpoint имеет типизированную функцию в `api/` и единый формат ошибок.

## План выполнения

### Фаза 0 — зафиксировать контракт и baseline

1. Сверить реальные ответы `GET /api/ancestries`, `GET/POST/DELETE /api/character` и auth API с backend-кодом/Swagger.
2. Зафиксировать типы запросов и ответов, сценарии `401`, `403`, validation error и network error.
3. Составить краткую матрицу старых маршрутов: переносимые, удаляемые, временные redirect.

Критерий готовности: новый frontend можно реализовывать без предположений о полях API.

### Фаза 1 — скелет и качество

1. Создать отдельный Vue 3/Vite проект, не меняя старый frontend.
2. Настроить TypeScript strict, path aliases, ESLint, Prettier и Vitest.
3. Подключить Vuetify 3, theme tokens и app shell.
4. Настроить единый Axios API-client и обработчик auth/error.

Критерий готовности: `lint`, `test`, `build` выполняются без предупреждений; новый проект открывает пустой dashboard в выбранном дизайне.

### Фаза 2 — доступ и навигация

1. Реализовать login/register и Pinia session store.
2. Реализовать route guards и redirect после входа.
3. Реализовать shell, mobile navigation, dashboard и empty state.

Критерий готовности: неавторизованный пользователь не попадает на защищённые маршруты; после входа видит dashboard.

### Фаза 3 — персонажи

1. Реализовать list, loading/error/empty states и переход в details.
2. Реализовать details и подтверждаемое удаление, если endpoint доступен.
3. Написать тесты API-преобразований, route guard и главных состояний списка.

Критерий готовности: пользователь может просмотреть список, открыть карточку и удалить своего персонажа с корректной обратной связью.

### Фаза 4 — wizard создания персонажа

1. Реализовать типизированную загрузку родословных.
2. Построить 4-шаговый flow и локальное состояние draft-а.
3. Реализовать выбор free boosts с явным числом слотов и запретом недопустимых комбинаций.
4. Добавить экран проверки перед отправкой, API-validation errors и успешный переход в карточку/список.
5. Покрыть unit-тестами преобразование родословной, ограничения boosts и сборку create request.

Критерий готовности: пользователь создаёт валидного персонажа по текущим правилам MVP; ошибки сервера понятны и не теряют введённые данные.

### Фаза 5 — переключение

1. Ручная проверка сценариев в браузере на mobile и desktop.
2. Сравнить новый frontend с API-contract baseline.
3. Переключить интеграцию backend на новый build.
4. Удалить `pathfinder.vue/` и его устаревшие зависимости только после подтверждённого переключения.

Критерий готовности: production build нового frontend развёрнут, а в репозитории не осталось двух активных SPA.

## Осознанно отложено

- Store и inventory;
- перенос legacy Product/Category API;
- расширенная кастомизация внешнего вида;
- полный e2e-набор;
- миграция исторических данных frontend: в нём нет отдельного хранимого доменного состояния, которое требуется переносить.
